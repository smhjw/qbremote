## 2. EXISTING SPEED LIMIT FUNCTIONALITY

### API Endpoints (QbApi.kt, lines 193-204)
- POST api/v2/torrents/setDownloadLimit (limit in bytes)
- POST api/v2/torrents/setUploadLimit (limit in bytes)

### Data Models (QbModels.kt)
TransferInfo:
  - dl_rate_limit: Global DL limit (not exposed in UI)
  - up_rate_limit: Global UL limit (not exposed in UI)

TorrentProperties:
  - downloadLimit: Per-torrent DL limit (bytes)
  - uploadLimit: Per-torrent UL limit (bytes)

### Repository (QbRepository.kt, lines 307-316)
setTorrentSpeedLimit() calls both setDownloadLimit and setUploadLimit

### ViewModel (MainViewModel.kt, lines 1590-1594)
setTorrentSpeedLimit(hash, downloadLimitKb, uploadLimitKb)
  - Converts KB/s strings to bytes via parseLimitKbToBytes()
  - Calls repository.setTorrentSpeedLimit()

### UI (TorrentCards.kt, lines 612-653)
In TorrentDetailInfoTab composable:
  - Two OutlinedTextField inputs (KB/s)
  - Apply button
  - Wrapped in: if (capabilities.supportsPerTorrentSpeedLimit)

### Strings (3 locale files)
  detail_section_speed_limit (CN: "种子限速", EN: "Per Torrent Speed Limit")
  detail_download_kb_label (CN: "下载 KB/s", EN: "Download KB/s")
  detail_upload_kb_label (CN: "上传 KB/s", EN: "Upload KB/s")
  detail_action_apply (CN: "应用", EN: "Apply")

### Auto-Population (TorrentCards.kt, lines 203-208)
LaunchedEffect populates UI fields when detail properties load:
  - Converts bytes to KB/s: dl / 1024

### Capability Flag (BackendModels.kt)
supportsPerTorrentSpeedLimit = true for both qBittorrent and Transmission

## 4. API ENDPOINTS & ARCHITECTURE

qBittorrent API v2 paths:
  POST api/v2/auth/login
  GET api/v2/transfer/info (global speeds, limits, disk)
  GET api/v2/torrents/info (list with speeds)
  GET api/v2/torrents/properties (dl_limit, up_limit)
  POST api/v2/torrents/setDownloadLimit
  POST api/v2/torrents/setUploadLimit

Connection Settings:
  Host, Port, Username, Password, HTTPS flag, RefreshInterval, BackendType

Global Limits:
  - Exist in TransferInfo (downloadRateLimit, uploadRateLimit)
  - NOT exposed in app UI
  - Only per-torrent limits are managed

## 5. TORRENT DATA MODELS

TransferInfo (Global):
  downloadSpeed, uploadSpeed (current)
  downloadedTotal, uploadedTotal (cumulative)
  downloadRateLimit, uploadRateLimit (server limits - NOT in UI)
  freeSpaceOnDisk, dhtNodes, totalTorrentCount

TorrentInfo (Per-Torrent Current Stats):
  hash, name, category, tags
  downloadSpeed, uploadSpeed (current bytes/sec)
  downloaded, uploaded (cumulative bytes)
  ratio, state, progress (0.0-1.0)
  seeders, leechers, numComplete, numIncomplete
  addedOn, lastActivity, eta

TorrentProperties (Per-Torrent Settings):
  savePath, downloadLimit (bytes), uploadLimit (bytes)
  shareRatio, timeElapsed, seedingTime
  creationDate, pieceSize, comment, totalSize

## 6. DATA FLOW FOR SPEED LIMIT

UI Input (KB/s string)
  ↓
TorrentOperationDetailCard state (downloadLimitText, uploadLimitText)
  ↓
TorrentDetailPageContent callback (onSetTorrentSpeedLimit)
  ↓
MainViewModel.setTorrentSpeedLimit(hash, dlKb, upKb)
  - parseLimitKbToBytes() converts to bytes
  ↓
TorrentRepository.setTorrentSpeedLimit(profileId, hash, dlBytes, upBytes)
  ↓
QbRepository.setTorrentSpeedLimit(hash, dlBytes, upBytes)
  - executeWithRetry block
  - api.setDownloadLimit() and api.setUploadLimit()
  ↓
QbApi (Retrofit HTTP POST)
  ↓
qBittorrent API

## 7. KEY IMPLEMENTATION PATTERNS

Repository Pattern: QbApi → QbRepository → TorrentRepository → MainViewModel → UI

State Management:
  - ConnectionStore (DataStore): Persists profiles, settings
  - MainViewModel: Holds MainUiState
  - Local Composable State: downloadLimitText, uploadLimitText
  - RealtimeSpeedTracker: Speed history

Byte Conversion:
  Display: bytes / 1024 = KB/s
  Send: KB/s * 1024 = bytes
